Android SQLite 性能优化

发表于 2016-03-28   |   分类于 Android数据库   |   阅读次数


在android开发中遇到数据库需要并发访问的问题，这里通过源码分析SQLite，怎样高效使用SQLite

SQLiteOpenHelper

如其名，负责辅助数据库操作，提供了一些数据库生命周期回调方法，需要注意的是，因为这些生命周期回调是在getWritableDatabase或者getReadableDatabase时触发，所以这2个方法被建议不要在主线程操作

单个还是多个DbHelper

最常见对SQLiteOpenHelper的用法是

  1. 继承实现自己的DbHelper
  2. 在构造方法中设置数据库参数
  3. 在其生命周期回调中定制操作(建表，升级等)
  4. 在需要使用CRUD的地方通过DbHelper.getWritableDatabase或者DbHelper.getReadableDatabase获取SQLiteDatabase实例去操作数据。

可见都是调用了getDatabaseLocked这个方法，此处synchronized代码块其实已经完成了对SQLiteDatabase获取的加锁，也就是说在使用上面的put方法时，使用单个DbHelper并不需要加synchronized关键字。（这里有个例外，如果在操作方法中放入db.close，为了避免另一个线程使用已经关闭的数据库，还是需要在操作方法签名中加入synchronized关键字）那么在不同线程中使用多个DbHelper会怎么样呢？这里有个例子https://github.com/touchlab/Android-Database-Locking-Collisions-Example，在多个线程中使用多个DbHelper去做写操作时，异常android.database.sqlite.SQLiteDatabaseLockedException: database is locked (code 5)

开源库

1. GreenDao

号称比ORMLite性能更强，使用代码模板，依据Entity生成Dao，使得每个Entity有相应的Dao，使用起来比较麻烦。但是省略了反射去填充实体属性的步骤，性能较好，特别是表字段较多的情况下，使用了上面提到的全局使用一个db，并不重复打开和关闭，使用了缓存的SQLiteStatement来执行操作

2. ORMLite

核心代码基于JAVA，利用反射建立Dao以及填充Entity，使用只需要注册Entity的Class，比较简单
